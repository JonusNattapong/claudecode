Permission Model

How Claude Code balances safety with usability.

Permission Modes

ModeBehaviorUse Case
DefaultAsk before dangerous toolsProduction
AutoAuto-approve safe toolsQuick workflow
yoloLiteRead-only auto (Glob, Grep, Read)Exploration
yoloMost tools autoAutomation
yoloMaxIncluding destructive opsFull automation
yoloGodComplete autonomy + auto webFully autonomous

Permission Check Flow

1. User triggers tool (via AI request)
2. Get current permission mode
3. Check tool risk level (read/write/dangerous)
4. If allowed → execute
5. If blocked → prompt user
6. User approves/denies
7. Continue or abort

Safety Gates

// Security restriction gates
function checkSecurityRestrictionGate(gateName: string): boolean {
  // Check org policy, settings, etc.
  return !isBlocked(gateName)
}

// Usage
if (!checkSecurityRestrictionGate('tengu_disable_bypass_permissions')) {
  // Block bypass mode
}

YOLO Mode Extension

In this fork, we added YoloGod mode:

// YoloGod: Highest level of bypass
if (process.env.CLAUDE_CODE_YOLO_GOD === 'true') {
  // Skip ALL safety checks
  // Allow autonomous web search
}

Key Learnings

  1. Progressive bypass — Users can choose safety level
  2. Risk-based — Tools categorized by risk level
  3. Policy-aware — Organization settings can override
  4. Transparent — Users see what's being blocked
  5. Override-able — Can switch modes mid-session
Warning: YOLO modes bypass safety. Use with caution — especially YoloGod.
Pattern: The permission system shows how to balance security with usability — give users control while preventing accidents.
