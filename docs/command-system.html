Command System

How 104 slash commands are implemented.

Command Registration

src/commands.ts

All commands registered in one file:

// Import command modules
import agent from './commands/agent/index.js'
import git from './commands/git/index.js'
import model from './commands/model/index.js'
import mcp from './commands/mcp/index.js'
// ... 104 commands

// Register with command registry
export function registerAllCommands() {
  registerCommand('agent', agent)
  registerCommand('git', git)
  registerCommand('model', model)
  // ...
}

Command Structure

Each command follows the same pattern:

command-name/
├── index.ts        # Entry point (load function)
├── name.ts         # Command name
├── description.ts  # Help text
├── prompt.ts       # Interactive UI (JSX)
├── noninteractive.ts # CLI fallback (optional)
└── execute.ts      # Implementation (optional)
                   # (often inline in prompt.ts)

Command Types

1. JSX Commands (Interactive)

// Most common - returns React component
export const call: LocalJSXCommandCall = async onDone => {
  return <MyCommandUI onDone={onDone} />
}

2. Direct Commands (Non-interactive)

// Simpler commands - just execute
export async function call(args: string[]): Promise<CommandResult> {
  // Do work
  return { content: 'Done!' }
}

3. Hybrid Commands

// Interactive with fallback
export const call: LocalJSXCommandCall = async (onDone, args) => {
  if (args.length > 0) {
    // Non-interactive mode
    return executeDirect(args)
  }
  // Interactive mode
  return <InteractiveUI onDone={onDone} />
}

Key Learnings

  1. Lazy loading — Commands loaded on-demand via dynamic import
  2. JSX for UI — Interactive prompts built with React/Ink
  3. Callback pattern — onDone callback closes command
  4. Type-safe — TypeScript ensures command interface consistency
Pattern: The command system uses a plugin-like architecture — each command is independent and self-contained.
