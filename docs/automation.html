Automation Modes

Claude Code provides a progressive permission system with four automation tiers — from interactive approval to fully autonomous execution.

Mode Overview

ModeBehaviorUse Case
default Interactive approval for every tool execution Normal development with full visibility and control
yoloLite Read-only operations auto-approved; writes and shell commands prompt Code exploration, research, analysis in trusted repos
yolo Most tools auto-approved; dangerous operations still prompt Active development with reduced interruption in trusted repos
yoloMax Fully autonomous execution; bypass all permission confirmations Disposable sandboxes, CI/CD pipelines, fully trusted environments

Activating a Mode

Via Slash Command

The /yolo command opens an interactive mode picker:

/yolo

Select from the available modes using arrow keys and Enter.

Via Keyboard Shortcut

Cycle through modes directly from the chat input:

PlatformShortcut
macOS / Linux (modern terminal)Shift+Tab
Windows (without VT mode)Meta+M

Via CLI Flag

bun run src/main.tsx session --permission-mode yolo

Via Configuration

Set the default permission mode in ~/.claude/settings.json:

{
  "permissions": {
    "defaultMode": "yolo"
  }
}
Security Warning: Automation modes bypass permission prompts. Use yoloMax only in disposable sandboxes or fully trusted environments. The mode is not a substitute for proper sandboxing — combine with agents.defaults.sandbox.mode for non-main sessions.

YOLO Lite — Read-Only Automation

YOLO Lite is the safest automation tier. It auto-approves non-destructive operations:

  • Read — Reading files and directories
  • Glob — File pattern matching
  • Grep — Content search
  • WebFetch — Fetching URLs (read-only HTTP GET)
  • WebSearch — Search queries
  • CodeIndex — Codebase indexing and search

All write operations, shell commands, and git mutations still require manual approval.

# Activate via CLI flag
bun run src/main.tsx session --permission-mode yoloLite

YOLO — Standard Automation

YOLO mode extends YOLO Lite by auto-approving most development operations:

  • All YOLO Lite operations — Full read access
  • Edit / Write — File modifications within the working directory
  • Bash (safe commands) — Standard development commands (builds, tests, formatting)
  • Git (safe operations) — Status, diff, log, branch listing
  • NotebookEdit — Jupyter notebook edits

Dangerous operations still prompt, including:

  • Destructive git commands (push --force, reset --hard)
  • Commands outside the working directory
  • Shell commands matching dangerous patterns (rm -rf, curl | sh)
Classifier Integration: YOLO mode uses an AI classifier (YoloClassifier) to evaluate potentially dangerous bash commands. The classifier runs asynchronously and may auto-approve or block commands based on context analysis, with hard_deny rules that cannot be overridden without explicit user action.

YOLO MAX — Full Autonomy

YOLO MAX bypasses all permission checks. Every tool and command executes without prompting.

Use only when:
  • Running in a disposable container or VM
  • The repository is fully trusted and all changes are reversible
  • You understand that destructive operations will execute without confirmation
# Activate via CLI flag
bun run src/main.tsx session --permission-mode yoloMax

# Or set in settings.json
{
  "permissions": {
    "defaultMode": "yoloMax"
  }
}

Permission Rule System

Beyond mode selection, the permission system supports granular rules with three behaviors:

BehaviorEffect
allowAlways permit the tool, bypassing mode checks
denyAlways block the tool, regardless of mode
askAlways prompt for confirmation, even in automation modes

Rules are configured in settings.json with source tracking (userSettings, projectSettings, localSettings, cliArg, session):

{
  "permissions": {
    "allow": [
      "Bash(npm run test:*)",
      "Bash(git status:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(curl * | sh:*)"
    ],
    "ask": [
      "Bash(git push:*)"
    ]
  }
}

Additional Permission Modes

ModeBehavior
acceptEditsAuto-accepts file edits and reads; everything else prompts
bypassPermissionsLegacy alias for full bypass (similar to yoloMax)
dontAskSuppresses permission prompts where possible
planPlanning mode — tools are restricted to read-only for architecture discussion

Sandbox Integration

For non-main sessions (group chats, subagents), combine automation modes with sandbox isolation:

{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "non-main"
      }
    }
  }
}

This ensures automated subagents run in Docker containers even when the main session uses YOLO mode.

Status Display

The current permission mode is displayed in the prompt footer. Cycle modes with Shift+Tab (or Meta+M on Windows). The mode indicator changes color to reflect the current tier:

  • Default — Neutral color (interactive)
  • YOLO Lite — Blue (read-only auto)
  • YOLO — Yellow (standard auto)
  • YOLO MAX — Red (full auto)
