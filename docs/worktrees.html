Worktrees

Isolate parallel Clew sessions in separate git worktrees.

Git worktrees allow you to check out multiple branches of the same repository simultaneously. Clew uses worktrees to run parallel sessions without file conflicts — each session gets its own working directory and its own branch.

Why Worktrees?

  • Parallel agents — run multiple Clew sessions on different tasks at the same time
  • Conflict-free — changes in one worktree don't affect others
  • Isolated dependencies — each worktree can run bun install independently
  • Safe experimentation — try risky refactors without touching main

Usage

Starting a Worktree Session

# Create and enter a new worktree
clew --worktree fix-auth-bug

# The session starts in a new git worktree on a new branch
# All file changes stay isolated in that worktree

Configuration

FilePurpose
.worktreeincludeList of files/directories to symlink from the main repo into each worktree (e.g. shared configs)
.gitignoreAdd .claude/worktrees/ to your gitignore to exclude worktree metadata

Managing Worktrees

# List all git worktrees
git worktree list

# Remove a worktree when done
git worktree remove <path>

# Or let Clew clean up automatically
# (worktrees can be removed when the session ends)

Best Practices

  • One task per worktree — keep each worktree focused on a single feature or bug fix
  • Clean up when done — remove completed worktrees to save disk space
  • Use .worktreeinclude — for shared configuration files that all worktrees need
  • Rebase before merging — rebase worktree branches onto main before creating a PR

Limitations

  • Worktrees share the same git history — they diverge at the branch point
  • Large monorepos may benefit from sparse checkout to reduce disk usage
  • Not all VCS systems support worktrees (git only)
