Provider Pattern

How Claude Code routes AI requests to different providers.

The Problem

Different AI providers (Anthropic, OpenAI, Google, etc.) have different APIs, response formats, and capabilities. How do you build a unified interface?

The Solution: Provider Adapters

src/services/ai/providers/ProviderInterface.ts

Each provider implements a common interface:

interface Provider {
  // Streaming response (for real-time output)
  streamMsg(
    messages: Message[],
    options: StreamOptions
  ): AsyncIterable<ToolUse | ContentBlock>

  // Non-streaming response (simpler, one-shot)
  nonStreamingMsg(
    messages: Message[],
    options: NonStreamOptions
  ): Promise<Response>

  // List available models for this provider
  getModels(): Promise<Model[]>

  // JSON schema for tool results
  getToolResultSchema(): JSONSchema
}

Provider Registry

src/services/ai/providerRegistry.ts

Singleton registry manages all providers:

class ProviderRegistry {
  private providers = new Map<string, Provider>()

  register(name: string, provider: Provider): void
  get(name: string): Provider
  getAll(): Provider[]
  getDefault(): Provider
}

Provider Manager

src/services/ai/ProviderManager.ts

Orchestrates provider selection, API keys, model selection, streaming, and token usage:

class ProviderManager {
  // Select provider based on model name
  selectProvider(model: string): Provider

  // Route message to appropriate provider
  async streamMsg(messages: Message[]): Promise<AsyncIterable<any>>

  // Handle tool calls
  parseToolCalls(response: Response): ToolCall[]

  // Track token usage
  trackUsage(usage: Usage): void
}

Model Discovery

src/services/ai/providerModels.ts

Models are discovered and cached for 5 minutes:

// Get models from all configured providers
const models = await getProviderModels()

// Result cached for 5 minutes
// Returns: [{ provider: 'anthropic', models: [...] }, ...]

Key Learnings

  1. Interface-based design — Common contract enables swapping providers
  2. Registry pattern — Singleton manages provider lifecycle
  3. Model routing — Parse model name (e.g., "openai/gpt-4o") to select provider
  4. Caching — Model discovery cached to reduce API calls
  5. Streaming support — Both streaming and non-streaming interfaces
Pattern: This is a classic Adapter pattern combined with Registry — great for extensible systems.
