Plugin Development

Create, package, and publish custom plugins

TL;DR Create a folder with a plugin.json manifest, add your commands/agents/skills/hooks, then install via /plugin install path/to/plugin or publish to a marketplace.

Plugin Directory Structure

my-plugin/
├── plugin.json              # Manifest (metadata + capabilities)
├── commands/                # Custom slash commands (markdown files)
│   ├── build.md
│   └── deploy.md
├── agents/                  # Custom AI agent configurations
│   └── test-runner.md
├── skills/                  # Reusable skill definitions
│   └── code-review.md
├── hooks/                   # Lifecycle hook configurations
│   └── hooks.json
├── output-styles/           # Custom output formatting
├── .mcp.json                # MCP server definitions (optional)
└── .lsp.json                # LSP server definitions (optional)

Manifest (plugin.json)

The plugin.json file declares metadata and all capabilities your plugin provides. All top-level fields except name are optional. Fields are merged with content found in the standard subdirectories (e.g., commands/). 

{
  // Required
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "Useful development tools",

  // Author info
  "author": {
    "name": "Your Name",
    "email": "you@example.com",
    "url": "https://github.com/you"
  },

  // Additional commands (beyond commands/ dir)
  "commands": [
    "./README.md"
  ],

  // User-configurable options (prompted at enable time)
  "userConfig": {
    "apiKey": {
      "type": "string",
      "title": "API Key",
      "description": "Your API key for the service",
      "sensitive": true
    }
  },

  // Plugin dependencies
  "dependencies": [
    "helper-tool@community-plugins"
  ]
}

Creating Commands

Place markdown files in the commands/ directory. Each file becomes a slash command named after the file (without extension). Frontmatter-style metadata is supported: 

# commands/build.md

Build the project using the configured build tool.
Runs the appropriate build command and reports results.

## Usage
/build or /build --watch

Commands can also be defined inline in plugin.json using the object format with source (file path) or content (inline markdown), plus optional metadata like description, argumentHint, model, and allowedTools.

Creating Agents

Agent files (agents/ directory) define specialized AI agent configurations with custom prompts, tool access, and model settings: 

# agents/test-runner.md

You are a test runner agent. Your job is to run tests and report
results. You have access to:
- BashTool (test execution only)
- ReadTool (reading test output)

Focus on:
1. Running the specified test suite
2. Collecting pass/fail results
3. Reporting clear summaries

## Settings
model: claude-sonnet-4-20250514
allowedTools: [Bash, Read]

Creating Skills

Skills (skills/ directory) are reusable task instructions that users can invoke during a session. Each skill is a markdown or structured file.

Hooks

Hooks allow your plugin to intercept Claudevil's lifecycle at specific points. Define them in hooks/hooks.json: 

{
  "hooks": {
    "PreToolUse": {
      "command": "node scripts/validate.js"
    },
    "PostPrompt": {
      "content": "Remember to run tests after editing."
    }
  }
}

Available hook points:

  • PreToolUse — Before a tool executes
  • PostToolUse — After a tool completes
  • PreBash — Before a bash command runs
  • PostPrompt — After the system prompt is built
  • PreAcceptEdit — Before an edit is applied

MCP & LSP Servers

Plugins can bundle MCP (Model Context Protocol) and LSP (Language Server Protocol) servers. Define them in the manifest or in dedicated files: 

// In plugin.json
{
  "mcpServers": {
    "my-service": {
      "command": "node",
      "args": ["server.js"],
      "env": {
        "TOKEN": "${user_config.apiKey}"
      }
    }
  }
}

User Config

The userConfig field declares values the plugin needs from the user. At enable time, users are prompted for each field. Non-sensitive values are stored in settings.json; sensitive values go to secure storage (macOS keychain or .credentials.json).

Available config types: string, number, boolean, directory, file.

Values are accessible as ${user_config.KEY} in MCP/LSP server config, hook commands, and skill/agent content.

Channels

Channels allow your plugin to inject messages into the conversation from external services (Telegram, Slack, Discord, etc.). Each channel binds to an MCP server that emits notifications/claude/channel notifications: 

{
  "channels": [{
    "server": "telegram-bot",
    "displayName": "Telegram",
    "userConfig": {
      "botToken": { "type": "string", "sensitive": true, ... }
    }
  }]
}

Testing Your Plugin

Validate your plugin manifest:

claude plugin validate ./my-plugin

Install from a local path:

claude plugin install ./my-plugin

Publishing to a Marketplace

To share your plugin, add it to a marketplace manifest. Marketplaces are defined in marketplace.json files hosted on GitHub, npm, or your own server: 

{
  "name": "community-plugins",
  "owner": { "name": "Community" },
  "plugins": [
    {
      "name": "my-plugin",
      "source": {
        "source": "github",
        "repo": "you/my-plugin"
      },
      "description": "Useful dev tools"
    }
  ]
}
