Memory System

How Claude Code remembers user preferences and project context across sessions, with semantic search and cross-lingual recall.

Memory Types

TypeScopeDescription
userprivateUser's role, goals, preferences
feedbackprivate/teamGuidance from user about how to work
projectteamProject-specific conventions
referenceteamExternal references, docs

Memory Files

.claude/
└── memory/
    ├── MEMORY.md              # Main entry point (auto-loaded)
    ├── user/                  # User memories
    │   ├── role.md
    │   └── preferences.md
    ├── feedback/              # Feedback memories
    │   └── testing-guidance.md
    ├── project/               # Project memories
    │   └── conventions.md
    └── reference/             # Reference memories
        └── api-patterns.md

How It Works

1. Loading

At session start, MEMORY.md is loaded into the system prompt:

// src/memdir/memdir.ts
await loadMemoryPrompt()
// Returns formatted memory content
// Included in system prompt automatically

2. Finding Relevant Memories

Claude Code uses a hybrid approach to find relevant memories:

  • Semantic Search (fast, local, cross-lingual) — Uses embeddings to find memories by meaning
  • LLM Selection (fallback) — Uses AI to select memories based on context
// src/memdir/findRelevantMemories.ts
const memories = await findRelevantMemories(query, memoryDir, signal)
// First tries semantic search, falls back to LLM if needed

3. Semantic Search

The semantic search system uses @xenova/transformers with a multilingual embedding model:

  • Model: ibm-granite/granite-embedding-97m-multilingual-r2
  • Languages: 100+ (including Thai, Chinese, Japanese, Korean)
  • Size: ~25MB (quantized)
  • Speed: Fast CPU inference
// src/memdir/semanticSearch.ts
import { searchMemories } from '../memdir/semanticSearch.js'

// Cross-lingual search works automatically
const results = await searchMemories("ระบบล็อกอินมีปัญหา", 5, 0.6)
// Returns: [{ file, score, content, ... }]

// Or search in English, find Thai memories
const results = await searchMemories("login authentication issue", 5, 0.6)
// Still finds "ระบบล็อกอินมีปัญหา" because embeddings share the same space

4. Memory Detection

Detects if file operations target memory files for special handling:

// src/utils/memoryFileDetection.ts
isMemoryFile(path): boolean      // Is this a memory file?
isMemoryDirectory(path): boolean // Is this memory dir?
isMemorySearch(toolInput): boolean  // Is search targeting memory?

Semantic Search Architecture

How It Works

User Query: "แก้ bug ที่ login page"
       ↓
[Embedding Model]
granite-97m-multilingual-r2
       ↓
Query Vector: [0.12, -0.45, 0.88, ...]
       ↓
[Cosine Similarity]
Compare with all memory embeddings
       ↓
Results (sorted by relevance):
1. auth-flow.md (92%)
2. testing-preference.md (78%)
3. user-role.md (65%)

Embedding Cache

Embeddings are cached as .embedding.json files alongside memory files to avoid recomputing on every search:

.claude/
└── memory/
    ├── auth-flow.md
    ├── auth-flow.md.embedding.json    ← Cached embedding
    ├── testing-preference.md
    └── testing-preference.md.embedding.json

Cache is automatically invalidated when memory files are modified.

Hybrid Search Flow

1. User query comes in
   ↓
2. Try semantic search (fast, local)
   ├── If found (score > 0.6) → Return results
   └── If not found → Fall back to LLM
       ↓
3. LLM selection (slower, costs API)
   └── Return AI-selected memories

Memory Frontmatter

Each memory file can have frontmatter metadata:

---
type: user
owner: private
description: User's preferred testing framework
---

# Testing Preferences

User prefers integration tests over unit tests for auth flows.
Always use bun test instead of jest.

Auto Memory

Claude Code can automatically remember important context during conversations:

// settings.json
{
  "autoMemoryEnabled": true,
  "autoMemoryDirectory": "~/.claude/projects/myproject/memory/"
}

Auto memory is enabled by default. To disable:

# Environment variable
export CLAUDE_CODE_DISABLE_AUTO_MEMORY=true

# Or in settings.json
{
  "autoMemoryEnabled": false
}

Key Learnings

  1. Memory is separate from context — Loaded at session start, not part of rolling window
  2. Type-based organization — user, feedback, project, reference
  3. Scoped access — private vs team memories
  4. Frontmatter metadata — type, owner, tags, description
  5. Auto-loading — MEMORY.md loaded automatically at session start
  6. Detection — Can detect memory file operations for special handling
  7. Semantic search — Cross-lingual recall using embeddings (100+ languages)
  8. Hybrid approach — Fast local search + LLM fallback for complex queries
  9. Embedding cache — Cached as .embedding.json files, auto-invalidated on change
  10. Model — IBM Granite 97M multilingual (~25MB quantized, fast CPU inference)
Tip: Semantic search works across languages. You can search in English and find memories written in Thai, Chinese, or any other supported language.
Note: The first semantic search may take a moment to download the embedding model (~25MB). Subsequent searches are fast as the model is cached.
