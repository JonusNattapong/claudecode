Permission Approval Flow

When the AI requests a tool execution, it goes through this pipeline:

  1. Permission mode check — The current mode determines the base behavior (ask, auto-deny, plan-only, etc.).
  2. Allow/deny rule matching — Pattern-based rules from user, project, local, flag, and policy settings are evaluated. Later sources override earlier ones.
  3. Shell command analysis — Bash and PowerShell commands are statically analyzed for destructive patterns (rm -rf, git push --force, etc.).
  4. PreToolUse hooks — Custom hook scripts can inspect, modify, allow, or deny the tool call. Hooks can return allow, deny, or defer.
  5. User prompt — If not auto-allowed, the user sees a permission dialog with command details and risk assessment.
  6. Execution — The tool runs (possibly sandboxed), with output captured for the AI.
  7. PostToolUse hooks — After execution, hooks can inspect results, log activity, or modify the output.

Sandboxed Bash Tool

The Bash tool supports sandboxed execution that restricts what commands can do:

Linux Sandboxing

  • PID namespace isolation — Subprocess runs in its own PID namespace, preventing it from seeing or signaling host processes.
  • Network restrictions — Configurable network access control. Auto and bypass-permissions modes can be configured for sandbox network access.
  • macOS sandbox — allowMachLookup support for controlled inter-process communication.

Command Safety Checks

CheckWhat It Catches
Destructive patternsrm -rf, git push --force, git reset --hard, git clean -fd
CWD-changing operationscd, pushd, popd, cd.., cd\, bare drive letters (D:, C:)
Env var bypassesCommands that manipulate $PATH, $HOME, or other security-sensitive variables
Shell expansions$VAR and $(cmd) patterns in sandbox auto-allow contexts
Redirect bypassesOutput redirection that could overwrite protected files

PowerShell Security

Windows PowerShell commands receive equivalent hardening:

  • CWD detection — cd.., cd\, cd~, and bare drive letters correctly detected as directory-changing operations.
  • Permission bypass prevention — Parser-level shortcuts cannot bypass the cd+compound guard in permission checks.
  • Separate tool — PowerShellTool alongside BashTool with its own security analysis.

Allow/Deny Rule Patterns

Create granular rules to control specific tools:

# Allow all git operations
Bash(git *)

# Block destructive operations
Bash(rm *)

# Allow only specific npm commands
Bash(npm run *)

# Allow reads in a specific directory
Read src/**/*.ts

# Block writes outside the project
Write !../**

Security Best Practices

  • Start with default mode — prompts for sensitive operations. Only escalate when needed.
  • Use allow rules, not bypassPermissions — rules are auditable and scoped. Bypass removes all checks.
  • Add project-level deny rules for production repos to block destructive commands.
  • Audit with PostToolUse hooks — log all tool executions for security review.
  • Keep disallowedTools updated as new tools are added to the system.
